# tbd — The Builders Depot dbt Wizard Hands-on Lab

Welcome to **tbd — The Builders Depot**. Yes, `tbd` is `dbt` backwards; we are absolutely leaning into the bit.

The Builders Depot is a home-improvement retailer with customers, orders, products, inventory, promotions, support tickets, and multiple store/warehouse locations. This repo powers a dbt Wizard hands-on lab where attendees join the analytics engineering team and help the business answer realistic questions from Operations, Marketing, and Data Platform stakeholders.

The point of the demo is not just to show that Wizard can write SQL. The story is that a new analytics engineer can start with a messy business question, discover the right dbt assets, inspect real data, understand grain and lineage, make a safe change, and validate the result without getting lost in the repo.

## The story we are telling

The Builders Depot sells everything a home-improvement customer might need: tools, fixtures, paint, lumber-adjacent supplies, garden gear, hardware, and seasonal promo items. The business is growing quickly, and the analytics team is trying to keep the data platform sturdy while teams ask more operational questions:

- **New analytics engineer onboarding:** "I just joined The Builders Depot. What does this dbt project do, where are the marts, and how do I ship something small safely?"
- **Inventory misallocation:** "Operations thinks inventory was misallocated across store/warehouse locations before a sale. Which locations are over-counted or under-counted?"
- **Support-ticket context on orders:** "The Director of Operations wants to know which orders generated support burden. Can we enrich orders with ticket counts without breaking downstream marts?"
- **Product source schema drift:** "A product source column changed overnight and the product lineage is red before standup. Can we diagnose the break, preserve the public dbt contract, and get the pipeline green?"
- **Marketing segmentation:** "Marketing wants a reusable campaign audience of VIPs, big spenders, and category-loyal customers based on recent purchase behavior by store."

Together, those scenarios show the everyday analytics engineering loop: orient, inspect, model, preview, validate, and only then materialize when it is safe.

## Project shape

- **Adapter:** Snowflake
- **dbt profile:** `dbt_hands_on_lab_aws`
- **Business domain:** home-improvement retail analytics for The Builders Depot
- **Staging models:** clean and standardize raw retail sources from Snowflake
- **Intermediate models:** enrich orders, customers, products, inventory, support tickets, and store/warehouse context
- **Marts:** table materializations for core analytics, Operations, and Marketing use cases
- **Tests:** schema tests and custom assertions for revenue, inventory, and relationships

Schemas are generated by layer:

```text
<target_schema>_staging
<target_schema>_intermediate
<target_schema>_marts
<target_schema>_marketing
```

## Common commands

From the project root:

```bash
dbt compile
dbt build --target dev
```

For Wizard-driven lab work, prefer asking Wizard to inspect models, preview data, and make changes rather than editing SQL by hand.

## Guided scenario skills

Repo-local Wizard skills live in `skills/`. This branch intentionally ships a single consolidated skill so every attendee gets the same lab flow:

- `lab` — full 15-minute guided hands-on lab with quiet setup and all five attendee prompts

Install or refresh the repo-local skills with:

```bash
python3 lab_reference/lab_scripts/install_skills_from_branch.py
```

After installing skills, restart Wizard so it picks them up.

## Reset safety

The protected raw source schema is:

```text
SNOWFLAKE_SUMMIT_2026_HOL_DB.SF_HOL_2026_RETAIL
```

Do **not** drop this schema during normal lab resets. Use the reset guides in `lab_reference/` so we only remove the current attendee's dbt-created schemas and preserve shared source data.
