## Context

The current DML has two structural problems causing repeated bugs:

1. **Two-phase product inserts** — 13 "scenario-critical" products inserted via VALUES, then 487 bulk products via GENERATOR with a `NOT IN` exclusion list. This caused duplicate products, required fragile ID filtering, and uses different code paths for the same table.

2. **Two-phase customer inserts** — 15 scenario C customers via VALUES, then 455 bulk customers via GENERATOR with a `NOT BETWEEN` exclusion. Same class of bug (GENERATOR+WHERE filtering).

The root cause: mixing VALUES-based inserts with GENERATOR-based inserts for the same table forces ID exclusion logic that Snowflake's GENERATOR doesn't handle reliably inline.

## Plan: Single-pass generation with scenario overrides

Restructure the DML into two clean phases with no ID conflicts:

### Phase 1 — Base data generation (all 500 products, all 470 customers in single passes)

**Products**: One single `INSERT...SELECT` that generates all 500 products in one statement. Use a CASE expression on the product_id to apply scenario-specific values (name, SKU, subcategory, vendor_id) for the 13 critical IDs, and default generated values for everything else. No VALUES block, no exclusion filter, no second insert.

```sql
INSERT INTO RET_PRODUCTS (...)
WITH ids AS (
    SELECT seq4() + 1 AS product_id FROM TABLE(GENERATOR(ROWCOUNT => 500))
)
SELECT
    'TBD_PR_' || LPAD(product_id::VARCHAR, 6, '0'),
    product_id,
    -- SKU: CASE WHEN product_id = 10 THEN 'PWR-DRL-010' WHEN product_id = 42 THEN ... ELSE <generated> END,
    -- NAME: CASE WHEN product_id = 10 THEN 'DeWalter 20V MAX...' WHEN ... ELSE <generated> END,
    -- SUBCATEGORY_ID: CASE WHEN product_id = 10 THEN 1 WHEN product_id = 42 THEN 2 WHEN ... ELSE MOD(...) END,
    -- VENDOR_ID: CASE WHEN product_id IN (101,102,106) THEN 1 WHEN ... ELSE NULL END,
    ...
FROM ids;
```

**Customers**: One single `INSERT...SELECT` generating all 470 customers. Use CASE for the 15 scenario C customers (401-415) to set exact names/addresses, default generated values for everything else. No VALUES block, no exclusion filter, no second insert.

```sql
INSERT INTO RET_CUSTOMERS (...)
WITH ids AS (
    SELECT seq4() + 1 AS cid FROM TABLE(GENERATOR(ROWCOUNT => 470))
)
SELECT
    'TBD_CU_' || LPAD(cid::VARCHAR, 6, '0'),
    cid,
    -- FIRST_NAME: CASE WHEN cid = 401 THEN 'Mike' WHEN cid = 402 THEN 'Sarah' ... ELSE <generated> END,
    -- ADDRESS: CASE WHEN cid = 401 THEN '1 VIP St' ... ELSE cid || ' Hardware Way' END,
    ...
FROM ids;
```

All other seed tables (warehouses, categories, subcategories, promotions, etc.) stay as VALUES — they're small, static, and don't use GENERATOR.

### Phase 2 — Scenario data (appended at the bottom, after all base data)

No changes to structure — the scenario sections (A: shipments/inventory, B: vendors/orders/returns, C: orders/order items) already work correctly and don't have GENERATOR filtering issues. They stay at the bottom of the file as-is.
