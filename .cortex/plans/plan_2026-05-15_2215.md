## Implementation Steps

**Approach:** Query the current validated test schema to extract the exact data that was produced, then write it as hardcoded VALUES. This guarantees the hardcoded values match the validated output.

1. **Export all non-deterministic data from the test schema** — Run SELECT queries against each table in `TEST_DBT_WIZ_HOL_SCHEMA_RELOAD` to extract the exact rows as formatted VALUES tuples. Use a subagent to generate the SQL VALUES text for each table.

2. **Replace RET_PRODUCTS insert** (line 129) — Replace the single GENERATOR-based INSERT with `INSERT...VALUES` containing all 500 product rows. Split into chunks of 100 if needed for readability.

3. **Replace RET_CUSTOMERS insert** (line 312) — Replace with `INSERT...VALUES` containing all 470 customer rows.

4. **Replace RET_INVENTORY seed insert** (line 487) — Replace with `INSERT...VALUES` for the ~150 rows.

5. **Replace RET_INVENTORY_TRANSACTIONS insert** (line 522) — Replace with `INSERT...VALUES` for 150 rows.

6. **Replace RET_TICKETS insert** (line 572) — Replace with `INSERT...VALUES` for 50 rows.

7. **Replace RET_PRODUCT_REVIEWS insert** (line 625) — Replace with `INSERT...VALUES` for 150 rows.

8. **Replace Scenario A MERGE** (line 724) — Convert to a simple `INSERT...VALUES` for the 5 inventory rows (since on a clean run there's nothing to MERGE into — it's always an insert).

9. **Replace Scenario B orders** (line 768) — Replace GENERATOR with `INSERT...VALUES` for 475 rows.

10. **Replace Scenario B order_items** (line 794) — Replace CTE with `INSERT...VALUES` for 475 rows.

11. **Replace Scenario B returns** (line 833) — Replace CTE with `INSERT...VALUES` for ~99 rows.

12. **Replace Scenario C category-loyal orders** (lines 999-1071) — Replace 3 GENERATOR blocks with `INSERT...VALUES` for 33 rows.

13. **Replace Scenario C order_items** (line 1076) — Replace SELECT with `INSERT...VALUES` for ~82 rows.

14. **Replace all `CURRENT_TIMESTAMP()` with a fixed timestamp** — e.g., `'2026-05-15 12:00:00 -0700'::TIMESTAMP_TZ` — so `_FIVETRAN_SYNCED` is identical across runs.

15. **Regenerate `dml_test.sql`** from the updated `dml.sql`.

16. **Test** — Truncate test schema, run `dml_test.sql`, verify identical row counts and scenario results.
