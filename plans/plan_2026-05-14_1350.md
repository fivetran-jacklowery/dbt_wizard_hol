## Implementation Plan

Based on discovery, the schema has **three structural gaps** that require new tables and **two minor gaps** that require new columns on existing tables. Below is the full deliverable I will produce — a single comprehensive document covering all 5 phases.

### Context & Key Findings

**Existing schema:** 12 tables in `SF_HOL_2026_RETAIL` covering customers (500), orders (10K), order items (50K), products (2K), inventory (7.5K), inventory transactions (30K), warehouses (5), categories (12), subcategories (43), promotions (50), tickets (1K), reviews (5K).

**Date range:** Orders span 2023-11-13 to 2026-04-30. Current date context for "trailing 365/180 days" can anchor to ~2026-04-30.

**Critical gaps for scenarios:**
- **Scenario A:** No shipment/distribution-event table exists. Inventory transactions have `receipt`/`transfer` types but lack shipment-header structure.
- **Scenario B:** No returns table (just order status = 'returned'). No return-reason column. No vendor/supplier table (only `BRAND` on products).
- **Scenario C:** Orders have no `warehouse_id`/`location_id` FK. Categories exist (12) and satisfy the "3+ categories" requirement.

### Implementation Steps

1. **Write Phase 1 Discovery Summary** — Markdown tables showing schemas, row counts, date ranges, categorical values, entity mapping.

2. **Write Phase 3 Gap Analysis** — Per-scenario table (Requirement / Found in / Gap / Proposed change).

3. **Write Consolidated DDL** — New tables:
   - `RET_VENDORS` (Scenario B) — vendor entity with ID, name
   - `RET_RETURNS` (Scenario B) — line-level returns with reason, quantity, date, linking back to order_item + product
   - `RET_SHIPMENTS` (Scenario A) — shipment header with destination warehouse, date
   - `RET_SHIPMENT_LINES` (Scenario A) — shipment line items with product, quantity
   
   New columns on existing tables:
   - `RET_PRODUCTS.VENDOR_ID` (Scenario B) — FK to RET_VENDORS
   - `RET_ORDERS.WAREHOUSE_ID` (Scenario C) — FK to RET_WAREHOUSES (fulfillment location)
   
   All with `DEMO_BATCH_ID` marker columns on new tables.

4. **Write Consolidated DML** — Grouped by scenario:
   - **Scenario A:** Insert shipment + lines for 1 focal product → 5 warehouses (200 units each), then adjust inventory so 1 is short, 2 are over, 1 near-miss, 1 exact.
   - **Scenario B:** Insert 3 vendors, assign VENDOR_ID to ~10 products. Insert 50+ sales and 15+ defective returns for 3 "high defect" items. Insert near-misses (17-19%) and clean negatives (0-5%).
   - **Scenario C:** Insert/update orders with WAREHOUSE_ID. Plant 8-12 customers with transactions satisfying VIP, Big Spender, Category-Loyal, multi-segment, boundary, and negative cases.

5. **Write Validation Queries** — One per scenario with expected row counts.
